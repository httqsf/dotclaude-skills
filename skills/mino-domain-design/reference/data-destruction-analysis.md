# data-destruction-analysis — データ破壊駆動（Data Destroy Driven）

> 由来：本人が命名した「データ破壊駆動（Data Destroy Driven）」に対応。

## 核心
バグの多くは「壊れたデータ（欠損・不整合）」が原因だが、開発では正常系が優先され異常系対策が後手に回りがち。そこで **「クラッカーになったつもりで、どうやったらこのクラスのデータを破壊できるか」を能動的に考える**。破壊シナリオを列挙する過程で、データ完全性を守るために必要な制約（不変条件）が自動的に洗い出される。単なるデータ完全性を超え「ドメイン関連のデータとロジックに欠損・不整合がない＝**ドメインモデルの完全性**」を目指す。

## 分析手順
- **Step1 破壊シナリオ想定**: そのクラスの値をどう壊せるか列挙（例: 金額をマイナスに／通貨を不正変更／nullを混入）。
- **Step2 制約の明確化**: 破壊シナリオから不変条件を導出（amountは0以上／currencyは非null）。
- **Step3 実装**: 完全コンストラクタ＋`final` でカプセル化し、制約をコードに落とす（`invariant-modeling.md` 参照）。
- テストコードで異常値（境界値・null・負値）を必ず検証する。
- ドメイン単位でも破壊分析する（設計段階で異常系＝問題解決ルールを先に潰す）。

## 具体例・アンチパターン
- **注文数マイナス問題**: ショッピングサイトで注文数に `-1` を入力できると、後続処理で「返金指示」が発行される脆弱性が生じる。入力段階でモデルが弾いていれば防げた典型例。
- **メール設定モデル**の破壊分析で洗い出される制約: 「プライマリメールを削除されると困る」「同一メールを重複追加されると困る」「メールを全部削除されると困る」→ これらがそのままドメイン制約になる。
- 最終メッセージは「致命的障害はサービスの信頼と収益を損なう。今すぐデータ破壊駆動でモデルを点検せよ」という経営的危機感を伴う。
