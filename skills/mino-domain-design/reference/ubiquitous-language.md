# ubiquitous-language — ユビキタス言語／目的駆動名前設計

## 核心
ユビキタス言語を **「目的を表現する言葉」** と解釈する。同じ対象でも目的が違えば別の言葉（購入品／配送品）を使うことで関係者間の齟齬を防ぐ。これが書籍の中核概念 **「目的駆動名前設計」**（『良いコード／悪いコードで学ぶ設計入門』第10章）と直結。名前は読みやすさの問題ではなく **ロジック構造そのものを規定する** 設計行為。

（クラス/メソッド実装レベルの命名詳細は `mino-code-design/reference/purpose-driven-naming.md` を参照。ここではドメイン言語としての側面を扱う。）

## 実践指針
- 「存在ベース」ではなく「目的ベース」で名前を付ける（× 商品 → ○ 予約品／在庫品／注文品）。
- 可能な限り具体的で、意味範囲が狭く、特化した名前を選ぶ。
- ビジネス目的を名前として表現することで「関心の分離」を実現する。
- 会話・ドキュメント・コードで同じ言葉を一貫して使う（設計と業務仕様の乖離を防ぐ）。
- 適切な名前が付けられないときは、対象を正しく認識できていないサイン（＝モデリング不足）と捉える。

## 具体例
「商品」という広すぎる名前は、あらゆるロジックを引き寄せて低凝集な巨大クラスを生む。関心事が「在庫」なら `在庫品`、「予約」なら `予約品` と命名するだけで、そのクラスに集まるべきロジックが自然に絞られ、凝集度が上がる。
