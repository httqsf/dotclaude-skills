# bounded-context-discovery — 境界づけられたコンテキストの発見

## 核心
境界づけられたコンテキストを「技術的な分割単位」ではなく **「ユーザーとその目的の違いで引く境界線」** として捉える。同じ「商品」でも「買う人／配送する人／在庫管理する人」で目的が異なり、目的が違えば別モデル・別コンテキストとして分けて作るべき。DDDそのものを「利益を出せるソフトを作るために当然やるべきこと＝ビジネス都合を考慮したアーキテクチャ設計」と位置づける。

## 実践指針
- まず「登場するユーザー（アクター）は誰か」「各ユーザーの目的は何か」を洗い出す。目的は必ず人が決めている。
- 「このユーザーと、このユーザーの目的は違う」という切れ目を見つけ、その範囲を1つのコンテキストとする。
- 同一物理概念（例: 商品）を、目的ごとに別概念（購入品／配送品／在庫品）へ解釈し直す。
- 目的ごとにシステム（モデル）を分けて作る。1つの巨大モデルに全目的を詰め込まない。
- 些細な目的差（予約注文・サブスク注文・抽選注文など）もフラグで吸収せず、目的単位で分離する。

## 具体例
通販アプリの「商品」を単一クラスにすると、ID・名前・仕入単価・在庫数・審査結果まで混在した巨大クラスになる。目的で切ると「審査品目」「在庫品」「注文品（購入品／配送品）」という別コンテキスト・別モデルに分割される。
