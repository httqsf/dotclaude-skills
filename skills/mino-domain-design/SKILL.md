# mino-domain-design — ミノ駆動流ドメインモデリング

ドメイン（問題領域）を、**目的＝見えない概念を軸に**モデル化する。物理的に見えるもの（犬・猫・商品・ユーザー）に引っ張られず、目的・問題・出来事・関係性を顕在化させる。

> 前提：`mino-design-core` の「目的→課題→手段」の連鎖。5トピックはすべて「**目的（＝見えない概念）を軸に設計する**」で一貫する。

## 貫く思想
境界づけ＝目的で分ける／命名＝目的を名前にする／invisibleモデリング＝目的や出来事を可視化／不変条件＝目的上あってはならない状態を禁止／データ破壊駆動＝目的を破壊する手口を先読み。合言葉は「**目を閉じ、心眼を開け**」。

## いつ・どう使うか（モデリングの手順）

1. **アクターと目的を洗い出す** → 誰が使うか、各ユーザーの目的は何か（`bounded-context-discovery`）
2. **目的ごとに言葉を与える** → 「商品」ではなく「注文品／在庫品／配送品」（`ubiquitous-language`）
3. **見えない概念を炙り出す** → 目的・問題・出来事・関係性を可視化（`invisible-driven-modeling`）
4. **不変条件で守る** → 値オブジェクト＋完全コンストラクタで「常に正しい状態」を保証（`invariant-modeling`）
5. **破壊シナリオで検証** → 「どう壊せるか」を考え、制約を洗い出す（`data-destruction-analysis`）

クラス/メソッドレベルの実装は `mino-code-design`、既存モデルの分割は `mino-refactoring` へ。

## クイック原則
- 物理存在をそのままクラス化しない（巨大 `User` クラスは目的特化モデルへ分解）
- 些細な目的差もフラグで吸収せず、目的単位で分離する
- **異常系ファースト**（異常系＝問題解決ルール＝条件分岐の宝庫）
- 出来事（注文→請求→キャンセル）は個別モデル化（イミュータブルデータモデリング）
- 関係性は交差エンティティでモデル化し Null許容カラムを排除
