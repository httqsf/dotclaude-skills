# ai-assisted-refactoring — AI支援リファクタリング

## 核心
「**AIの有無に関係なく、正しくリファクタリングするには変更容易性の設計スキルがどうやっても必要**」。AIは可読性の悪いコードでも高速に意図を推論する「コードリーディング／パターン認識」が得意だが、**何が負債で、どこからどう直すかの戦略的方向付けは人間の責務**。AIは分析・テスト生成・新規作成の加速に使い、名前変更やメソッド抽出などの機械的操作はIDE機能に任せる。

## 実践指針（役割分担）
- **名前変更・メソッド抽出はAIにやらせず、エディタ／IDEのリファクタリング機能で行う**（トークン浪費と誤操作を避ける）。AIには「目的の分析」と「より目的が明確な名前の提案」をさせる。
- **テストコード生成はAIに任せる**。「テストコードの実装はもはや人間の仕事ではない」。事前条件・事後条件・不変条件をGiven-When-Then構文で書かせ、agent skill化して再利用する。
- **AIは複雑コードの改変より"新規作成"が得意**。ストラングラーフィグパターンで旧モジュールを新モジュールに段階的置き換えする。
- **静的解析の限界をAIで超える**。従来の静的解析は行数・重複など機械的指標に限られるが、AIは「コードの意図」を推論しカプセル化や関心の分離を高精度に判定できる（自作MCPサーバー「モディフィウス」は400行の分析を約20秒＝人間の40倍以上）。
- **ドメインモデルには必ず「制約」を設計する**。箱とデータだけ作らず、Value Objectで「マイナス注文数」等の不整合を防ぐ制約を組み込む（`mino-domain-design` 参照）。

## 手順（AIリファクタリングのステップ）
1. AIに「このコードの目的を分析して」と指示し意図を推論させ、目的が明確な名前を提案させる（実行はIDE）。
2. AIに事前・事後・不変条件を検証するテストコードを生成させる。
3. 副作用（I/O・インスタンス変数変更）を切り離し純粋関数を抽出（候補列挙はAI、抽出はIDEのメソッド抽出、テスト追加はAI）。
4. **目的別クラス分離（シンプルながら強力）**: 複数目的が混在するクラスを目的ごとに丸ごとコピー → 各コピーから無関係な処理を削除 → 共通処理は独立クラスに抽出して呼び出す。DMMでの実績あり。
5. イベントストーミングで集約（ドメインモデル）を導出し、ストラングラーフィグパターンで段階的に置き換え。

## 実例（DMMユーザーレビューグループ, 2025）
社内「ミノ駆動設計講座」で学んだ「関心の分離」を土台に、AIエージェント（Devin）で3,000行超のInfra層を①テーブルごとのリポジトリ分割 ②ドメイン層へのロジック移譲（VO・サービス化）③GORM v2移行 へリファクタ。**新旧Infra層を並行運用しInterfaceで差し替え、問題時に即ロールバック**できる体制で「外部仕様を変えず中身だけ整理」。工数は1人月以上→1ファイル1人日以下に半減、PR作成数は約3倍。
