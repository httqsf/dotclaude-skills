# legacy-purpose-split — レガシーコードの目的分割

## 核心
数千〜数万行に膨れた巨大クラス（神クラス／大きな泥団子）は、複数のユースケースと密結合し変更困難を生む。解決の要は**「関心の分離」と「役割駆動設計」**。神クラスの根本原因は「対象の細かな違いを見分けられていない（解像度が低い）」ことにあり、**モノ（ユーザー・商品・金額）ではなく、コンテキストごとの"役割／目的"でクラスを分割する**ことが本質。

## 実践指針
- **「モノ」で命名しない**。「ユーザー」「商品」「金額」といった意味範囲の広い名前は"あらゆるロジックを引きつける心理的引力"で巨大化を招く。「出品者」「購入者」「請求金額」など**役割・目的を表す特化した狭い名前**を付ける。
- **コンテキスト（場面）を洗い出し、役割ごとにクラスを分ける**。同じ「自動車」でも配送コンテキストと販売コンテキストで関連属性は異なる。
- **各クラスは最小限のフィールド・メソッドだけを持つ**。不要な情報へのアクセスを排除し、疎結合・高凝集にする。
- **アンチパターンを避ける**: ①技術駆動命名（Int〜/〜Flag）②ドメインモデル貧血症（〜情報/〜データ）③複数コンテキストの混在④仕様変更で意味範囲が変わる名前（「個人顧客」に法人属性が混入）。
- **共通化の判断は「コード重複」ではなく「ビジネス概念の重複」で行う**。概念が異なれば類似ロジックでも分離を保つ。

## 手順・実例
- 「商品」クラス →「注文品」「予約品」「発送品」へ関心分離。
- 単一「人クラス」（従業員・父親・ゲーム制作者が混在）→ 役割ごとに「従業員（機密情報）」「お父さん（家族情報）」「ゲーム制作者（制作知識）」へ分割。
- フリマの「ユーザークラス」→「出品者」「購入者」「発送者」「受取人」に細分化。動詞「出品する」をメソッドにするのでなく「出品」をクラスにする（動名詞でのクラス化）。

### 安全なリファクタリング手順（自爆奥義集の裏返し）
- 機能変更とリファクタリングを同時にやらない（最低でもコミットを分け、バグ修正後にリファクタ）。
- テストが無くて進められない場合は、IDEのメソッド抽出で小さく切り出し、段階的にテストコード化する。
- ロジックだけでなく**名前も必ずリファクタする**（名前が分割単位を左右する）。
- パフォーマンス最適化など「事情のあるコード」を調査せず強行しない。
