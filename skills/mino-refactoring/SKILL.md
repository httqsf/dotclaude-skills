# mino-refactoring — ミノ駆動流リファクタリング

技術的負債を「**変更容易性が低い（＝将来の変更コストが高い）構造**」、リファクタリングを「**未来の変更コストを低減する活動**」と再定義する。すべての起点は「**あるべき姿（理想構造）を先に定義する**」こと。理想を知らなければ負債は認識できない。

> 前提：`mino-design-core` の「目的→課題→手段」。目指す構造の作り方は `mino-code-design`／`mino-domain-design`。

## いつ・どう使うか（3ステップ）

1. **どこから直すか** → コアドメイン × 変更頻度で優先順位。効果の薄い安定部分は触らない（`debt-prioritization`）
2. **どう分けるか** → モノ（ユーザー・商品）でなく役割／目的でクラス分割（`legacy-purpose-split`）
3. **AIをどう使うか** → 戦略は人間、加速はAI。名前変更/抽出はIDE、分析/テスト生成はAI（`ai-assisted-refactoring`）

## クイック原則
- 「あるべき姿を先に定義する」— ゴール（機能性・変更容易性）なきリファクタは的外れ
- 「モノではなく役割／目的」「特化こそ正義」「解像度を上げる」
- 「戦略は人間、加速はAI」「テスト実装はもう人間の仕事ではない」
- **機能変更とリファクタリングを同時にやらない**（コミットを分ける）
- ロジックだけでなく**名前も必ずリファクタする**

## アンチパターン（リファクタリング自爆奥義）
- ほぼ変更されない安定部分を一生懸命リファクタする（費用対効果ゼロ）
- ビジネス概念が違うのに「コードが重複しているから」と無理に共通化する
- パフォーマンス最適化など「事情のあるコード」を調査せず強行する
- テストが無いまま大改造を始める
