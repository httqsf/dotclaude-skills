# evidence-gathering — 証拠収集（git履歴の扱いの正典）

## 核心

証拠は2系統——「**コードの実読**（統合点＝呼び出し・import・共有データ構造・DBアクセス・APIを直接読む）」と「**構造化ヒアリング**（ドメイン分類・チーム境界・既知の痛点・戦略方向）」。git履歴は警戒付きで扱う: **変動性の証拠にはしない**（コミット頻度は偶発的変動性＝設計の悪さを拾うため。公式資料にも "NOT commit history" と明記）。共変更分析は「**暗黙的結合の発見器**」＝強度側の証拠として使う。用途を取り違えないこと。

## 2系統の一次証拠

**コードからの証拠（→ 強度）**:
- 実際の統合点を読む: 関数呼び出し、import、共有データ構造、DBアクセスパターン、API呼び出し。Grep・Glob・LSP（参照検索・定義ジャンプ）で辿り、推測しない
- 構造（ディレクトリ・依存図）だけから強度を語らない。「実際に何の知識が共有されているか」まで下りる
- `docs/` の要件ドキュメントを先に読む（何がビジネス上重要か＝変動性の手がかりにもなる）

**ヒアリングからの証拠（→ 変動性・距離）**:
- 変動性: サブドメイン分類（core/supporting/generic）＋今後の移行・事業計画。これが変動性の一次証拠（履歴ではなくこちらが正）
- 距離: チーム所有境界・デプロイトポロジ・共有インフラ（組織構造はコード構造以上に実効距離を左右する）
- 痛点: 「変更が予想外に高コストな場所」「デプロイで壊れる場所」を最初に聞き、分析の焦点を決める

## git履歴・運用データの正しい読み方

本人の警告（Khononov、公式資料で一次確認）:
- **偶発的変動性**: コミットが頻繁でも、ビジネスドメインが変動的とは限らない。統合強度と距離の管理不全（劣悪な設計）が変更頻度を生んでいる可能性がある
- **偶発的非変動性**: 変更されていないコンポーネントが本質的に安定とは限らない。ビジネスは変えたいのに、リスクと工数が高すぎて諦めている可能性がある

この警告と整合する範囲での履歴データの用途（※以下4点は既存調査で整理した実務拡張であり、公式プラグインの手順には含まれない）:

```
git共変更分析（同一コミット/PRに現れるファイル組の集計）
  → 暗黙的結合・カスケード変更が実在する証拠（強度側）。変動性の測定器ではない
同時デプロイ要否・リリース順序調整の実態
  → ライフサイクル結合の証拠。分散モノリス（強×遠）の最有力な運用的兆候（距離側）
API変更時の追随修正クライアント数
  → 契約が知識を隠蔽できていない（contractに見えてmodel/functional）証拠（強度側）
仕様変更あたり関与チーム数
  → 組織的距離を越えたカスケードのコスト実証（距離側）
```

## 証拠→軸のマッピング表

```
統合点の実コード（呼び出し/import/DB/API）  → 強度レベル判定の一次証拠
重複実装の発見（同一ルールの多重実装）      → 暗黙的functional結合の証拠
git共変更分析                               → 暗黙的結合・カスケードの存在証明（強度側）
同時デプロイ・リリース調整の実態            → ライフサイクル結合・分散モノリスの証拠（距離側）
API変更時の追随修正数                       → 契約の知識隠蔽失敗の証拠（強度側）
仕様変更あたり関与チーム数                  → 組織的距離のコスト実証（距離側）
ドメイン分類ヒアリング＋事業計画            → 変動性の一次証拠（履歴ではなくこちらが正）
既知の痛点ヒアリング                        → 分析の焦点決定（どこを深掘りするか）
```

## 実践指針

- 変動性の一次証拠は常に「ビジネスドメイン分類＋戦略方向のヒアリング」。git履歴は補助・裏取りに格下げする
- git共変更分析は「変動性の測定」ではなく「暗黙的結合の発見器」として使う
- 安定して見えるコンポーネントには偶発的非変動性を疑う質問を1つ入れる（「ここは変えたいのに変えられていない、ということはないか」）
- 分散モノリス判定は運用証拠（同時デプロイ・リリース順序調整・障害の波及）で確定させる
- 痛点ヒアリングを先に行い、履歴分析はその周辺に絞る（全リポジトリの網羅分析はしない）
