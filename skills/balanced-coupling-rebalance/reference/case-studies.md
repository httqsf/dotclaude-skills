# case-studies — 典型ケースの再均衡化

## 核心

再均衡化の解は常に「API化・分割」ではない。**頻繁に共同変更されるものは距離を近づける（モノリスに戻す）方が正しい場合がある**。Khononovは分散モノリスを「ローカルな複雑性をグローバルな複雑性に交換しただけ（traded local complexity into global complexity）」と呼び、物理的分離が疎結合を意味しないことを繰り返し強調する（InfoQ、本人発言）。

## ケースA: 共有DB（侵入結合×サービス間距離）

評価: 侵入結合＋遠距離。スキーマが高変動なら不均衡。選択肢は4つ:

1. モジュラーモノリスへ統合して**距離を下げる**
2. DB所有権を1サービスに限定して**強度を下げる**（侵入→コントラクト）
3. 安定したイベント・API契約を公開して**強度を下げる**
4. 読み取り専用レプリカ・公開ビューで**強度を部分的に下げる**

本人の補足（InfoQ）: 「私的インターフェースへの依存を導入した時点で、実装詳細への依存になる」。ただし「レガシーシステムで誰も変更しないなら、DBから直接取って構わない」——**低変動が侵入結合すら正当化する例**。

## ケースB: 巨大共通ライブラリ

複数サービスが共有ライブラリのドメインモデルを直接利用するのは「モデル結合を遠距離に拡散」した状態。再均衡: 高変動の業務モデルは各コンテキストが自前で持ち（**強度を下げる**）、共通ライブラリにはログ・トレーシング・標準エラー形式など**低変動で汎用的な知識**（genericサブドメイン相当）だけを残す。

## ケースC: フロントエンド/バックエンド契約

DBモデルをそのままAPIレスポンスにするとFEが内部スキーマにモデル結合する。画面の意味に変換した契約（例: `displayName` / `membership` / `canPurchase` — TypeScript想定、他言語でも同等の考え方）で**強度を下げる**。ただし画面ごとの薄いAPI乱造は変換層を増やすだけ——「何の知識も隠蔽しないDTOはかえって複雑になる」という本人の警告が効く。どこまで契約を分けるかは画面仕様の変動性とチーム間距離で決める。

## ケースD: 同期API連鎖のマイクロサービス群

Order→Pricing→Campaign→Inventoryが同期APIで連鎖し、キャンペーン仕様変更のたびに全サービス同時リリースになるなら「距離だけを広げた密結合」＝機能結合×遠距離×高変動。再均衡の選択肢:

- (a) 同時変更されるサービス群をモジュラーモノリスに統合（**距離短縮**。「モジュラーモノリスはマイクロサービス分解の第一歩。同じ物理境界内なら間違いの修正がはるかに容易」——InfoQ、本人発言の趣旨）
- (b) 業務フロー知識を1サービスに集約しオーケストレーション化（**強度低下**）
- (c) 非同期イベント化でコントラクト結合に降ろす（**強度低下**）

本人由来の追加例（InfoQ）: 請求書ワークフロー（経費管理→承認→買掛金）で、各段階が下流の会計詳細を知らずに済む**コントラクト結合による段階的知識境界**の例。
