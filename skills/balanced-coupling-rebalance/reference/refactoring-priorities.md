# refactoring-priorities — リファクタリング優先順位

不均衡の**判定表の正典は `balanced-coupling-review/reference/imbalance-patterns.md`**（強×遠×高変動=Critical／強×遠×低変動=許容可能な技術的負債／弱×近=低凝集の疑い）。本ファイルはrebalance固有の「どの順で直すか」だけを扱う。

## 核心

すべての結合を弱めるのではなく、**「強い×遠い×高変動」を最優先**で是正する。公式reviewスキルは urgent（強×遠×高変動）／tolerable technical debt（強×遠×低変動）と優先度を区別する。同一クラス内に閉じた密結合（強×近）はバランスが取れており、緊急度が低い。

## 着手順

1. **優先度1: 強×遠×高変動**（分散モノリスの中核）。urgent。即時是正対象
2. **優先度2: 強×遠×中変動、中強度×遠×高変動**（既存調査の整理。書籍の記述粒度としては未確認——目安として扱う）
3. **監視に留める: 強×遠×低変動**は「許容可能な技術的負債」。台帳に記録して変動性の変化を監視する対象であり、即時是正対象ではない。低変動なら**侵入結合すら許容し得る**（「レガシーで誰も変更しないなら、DBから直接取って構わない」——InfoQ、本人発言）
4. **低優先: 強×近**（同一クラス・同一モジュール内の密結合）。1人が1PRで安全に変更できるならバランス済み。「疎結合にしたい」だけの理由でイベント・DI・抽象層を挟むと偶発的複雑性が増える
5. **弱×近**は低凝集の疑いとして距離を広げる候補に回す（判定は正典参照）

## 実践指針

- 是正候補はcriticalな「不均衡×高変動」ペアに集中し、些末な指摘を並べない（公式reviewスキルの制約の趣旨）。指摘の数ではなく**変更コストへの寄与**で選ぶ
- 変動性評価はGitコミット数だけでなくビジネス視点（core/supporting/generic分類）で行う。genericは「機能的変動性は低いが実装変動性は可変」（公式スキルの趣旨）。分類の正典は `balanced-coupling-volatility/reference/domain-classification.md`
- 是正の具体手段（どの軸をどちらへ）は `reference/which-axis-to-move.md`、監視対象に回した負債の再評価トリガーは `reference/rebalancing-triggers.md`

## 優先度マトリクス（要約）

```text
強×遠×高変動 → 最優先（urgent）
強×遠×低変動 → 許容可能な技術的負債（監視。侵入結合すら許容し得る）
弱×遠×高変動 → 契約と互換性管理があれば成立
強×近×高変動 → 共同変更しやすく合理的
弱×近       → 低凝集の疑い（距離を広げる候補）
```
