# rebalancing-triggers — 再均衡化の引き金

## 核心

設計は一度バランスを取れば終わりではない。「結合のバランスを評価しても、システムは変化していくので、その変化に気を配らなくてはいけない」「バランスが崩れたら、他の次元を調整して、バランスを回復させる必要がある」（第11章の趣旨、読者解説経由）。バランスを崩すのは主に**変動性の変化**（ビジネス側）と**距離の変化**（組織・デプロイ構造）であり、強度は自然には変わらず、放置により知識共有が増えて悪化する。

## 実践指針

- **変動性由来のトリガー**を監視する: ビジネスルールの変更頻度上昇（supporting→coreへの昇格）、新規事業領域化、競争優位領域化。「変動性を本当に評価するにはビジネスドメインを理解し、ビジネス戦略・競合との差別化要因を分析しなければならない」（InfoQ講演、本人発言）
- **距離由来のトリガー**を監視する: チーム分割・組織変更、モジュールの外部サービス切り出し、リポジトリ分割。距離は「共進化の社会技術的コスト」（公式README）なので、**コードが1行も変わらなくても組織変更だけでバランスは崩れる**
- **強度由来のトリガー**を監視する: モジュール肥大化（境界内に無関係な知識が蓄積）、API利用者の増加（契約変更コスト＝実効距離の増大）、同時変更（co-change）の増加
- Git履歴・障害履歴から「一緒に変更されるファイル/リポジトリ」「同時デプロイが必要なサービス」「1つの仕様変更に関与するチーム数」を定期観測する（書籍の趣旨に沿う運用化。手順自体は応用）
- 再均衡化のシグナルは「不均衡」単独ではなく「**不均衡×高変動**」の組み合わせで判定する（公式reviewスキルの警告対象の趣旨）
- `NOT VOLATILITY` で救済されていた不均衡（例:「レガシーで誰も変えないならDB直読みでも構わない」——InfoQ、本人発言）は、変動性が上がった瞬間に顕在化する。前提が崩れたら再評価が必須

## 具体例

成功したPoC（低変動コンポーネントのマイクロサービス抽出）を、他の高変動コンポーネントに横展開して失敗する企業のパターン（InfoQ、本人発言）。変動性の違いを無視した抽出が引き金の典型。
