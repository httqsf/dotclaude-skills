# tactical-ddd-connection — 戦術的DDDとの接続

## 核心

書籍第13章「均衡結合の実践」の解決策は戦術的DDD（値オブジェクト・集約・境界づけられたコンテキスト）に寄っている（読者解説: 「実際に用いるソリューションは、戦術DDDに偏った内容になっていた」）。ただし本書の位置づけはDDD入門ではなく、**「なぜその設計変更の意思決定をしたのか」を均衡結合モデルで説明する方法**——戦術的DDDのパターン群に対する「なぜその境界にしたのか」を説明する理論として機能する。

## 各パターンの3軸による読み替え

- **値オブジェクト**: 常に一緒に変わる複数の値（金額と通貨など）を1つの境界に閉じる＝**強×近**に再配置してバランスさせる装置
- **集約**: トランザクション整合性という強い知識共有を持つオブジェクト群を最小の距離（同一境界・同一トランザクション）に置き、集約間は結果整合＝弱い結合に落とす、という**強度×距離の均衡装置**
- **境界づけられたコンテキスト**: モデル結合を断ち切る単位。コンテキスト間はコントラクト結合（公開契約・変換層）に降ろすことで「遠い」ことを正当化する

（この読み替えは調査時の整理であり、本人の逐語表現ではない）

## 実践指針

- 「DDDパターンを適用しろ」ではなく「**このパターンを選んだ理由を強度・距離・変動性で説明できるか**」を問う。説明できない適用は再考する
- Khononov自身が『Learning Domain-Driven Design』（邦訳『ドメイン駆動設計をはじめよう』）の著者であり、公式プラグインでも変動性評価にサブドメイン分類（core/supporting/generic）をそのまま採用している。両著作は「サブドメイン分類＝変動性の代理指標」で接続されている。分類の正典は `balanced-coupling-volatility/reference/domain-classification.md`
- コンテキスト間契約の設計（強度をコントラクトまで降ろす）は `balanced-coupling-strength/reference/integration-strength-four-levels.md` の4段階を使う
- 集約の境界を決めるときは「一緒にコミットしなければならない知識か」（強度）と「別チーム・別デプロイに置けるか」（距離）を分けて評価する

## 概念の帰属（重要）

- 境界づけられたコンテキスト・集約・値オブジェクト・ユビキタス言語: **Eric Evans**（2003）が原典
- 集約設計の実践則（小さく保つ・IDで参照等）: **Vaughn Vernon** の貢献が大きい
- **Khononovの独自貢献**は、これらを「共有知識の量（強度）×距離×変動性」で再説明し、サブドメイン分類を変動性評価に接続した点。パターン自体の発明者ではない
