# avoid-over-mocking — テストダブルは極力使わない

## 核心
テストダブルは「本物を使えるものは本物を使う」が原則。t-wadaは段階戦略「**使わない → 作らない（既存のFakeを探す）→ 慎重に作る**」を提唱（この3段階は `t-wada-test-strategy/reference/test-ratio-migration.md` と共通。サイズ観点の詳細はそちらが正典）。テストダブルは忠実性を犠牲にするため、**責任境界（自組織の管理下にない依存、一番終端のプロセス外アクセス）にのみ**使うべきで、内部の責任境界に使うと「問題の発見を遅らせ、偽陰性を招く」。

## 指針
- テストダブルを**作る前に**考える。まず本物を使えないか、次に公式提供やOSSの**Fake実装**が無いか探す（自作は最後）。
- 使いどころは「一番終端のプロセス外にアクセスしているところ」＝**システム境界**。組織内部の境界には使わない。
- モックライブラリで「privateメソッドをこじあけてテストする」ゴリ押しは、リファクタリング耐性のないテストを生むアンチパターン。
- テストダブルの役割は2つ＝①テストしにくいものを**テスト可能にする**、②**テストサイズを下げる**（速度・決定性の向上）。
- 自組織内では、ダブルより本物同士を繋ぐ統合テストの方が退行保護・クロスチェック性が高い場合がある。
