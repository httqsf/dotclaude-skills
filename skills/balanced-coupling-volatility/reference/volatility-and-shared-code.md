# volatility-and-shared-code — 変動性と共有コード

## 核心

コードを共有するとは知識を共有すること。共有ライブラリは「モデル結合〜機能結合クラスの強い統合強度」を「遠距離（複数サービス・複数チーム）」に拡散させる装置になりうる。共有ライブラリは距離を縮めずに強度だけ上げるので、バランス式では `NOT VOLATILITY` だけが救済条項——したがって**共通ライブラリに置いてよいのは低変動・汎用の知識のみ**。高変動の業務モデル（コアドメインのモデル）を共有すると、モデルの変更が全利用者への連鎖変更として爆発する。

## モデルからの導出（本人の一次ソースより）

- 著者の公式プラグイン（機能結合の説明）: 極端なケースが**重複した知識（duplicated knowledge）**——同じビジネスルールがフロントエンドとバックエンドの両方に実装されている状態。仕様変更が両方に同時に反映されなければ不整合に陥る。共有か重複かに関わらず、同じ揮発性の知識を複数箇所が持てば連鎖変更が必然化する。
- Tech Lead Journal #188の本人の例: 暗号化アルゴリズムを複数コンポーネントに複製すると、セキュリティパッチのたびに連鎖変更が起きる。1つのモジュールにカプセル化して共有知識を最小化せよ。
- DDDの共有カーネル（shared kernel）＝モデル結合を意図的に許すパターン。Khononovの立場は、適用は同一チーム内など距離が近く、共有部分の変動を全員が即時に吸収できる場合に限る（距離と変動性で正当化できる場合のみ）。
- 書評経由のケーススタディ（二次）: 共有UIコンポーネントライブラリで、すべてのフロントエンド機能追加が2つのリポジトリの同時変更を要求する機能結合が発生——共有ライブラリが高変動の知識を含むと、距離を越えた連鎖変更が常態化する。

## 共有してよいもの／いけないもの

| 判定 | 例 | 理由 |
|---|---|---|
| 共有ライブラリに適する（低変動×汎用） | ログ出力、トレーシング、標準エラー形式、文字コード変換、安定した技術ユーティリティ | 汎用サブドメイン的知識で機能的変動性が低い。強度×遠距離でも `NOT VOLATILITY` でバランス成立 |
| 共有ライブラリに置くと危険（高変動×業務） | User/Order/Payment などのドメインモデル | 見かけは「再利用」だが実際は**強いモデル結合を遠距離に拡散**。コアドメインのモデルは定義上最も変動的で、変更のたびに全利用サービスの改修・同時リリースが必要＝分散モノリスの一形態 |

判定手順: (1) 共有したい知識のサブドメイン分類は？（→ `balanced-coupling-volatility/reference/domain-classification.md`） (2) その知識の変動性は（機能的/実装的に）？ (3) 利用者との距離は（同一チーム/別チーム/別リポジトリ）？ → 高変動×遠距離なら共有せず、各コンテキストが自前のモデルを持ち、統合はコントラクト（公表された言語・DTO・イベント）経由にする。

```typescript
// 危険: 共有ライブラリ @acme/shared-models にコアドメインのモデルを置く
// → 全サービスがOrderの内部構造（高変動の知識）に結合し、変更のたびに同時改修
import { Order } from "@acme/shared-models";

// 良い: 各コンテキストが必要最小限の自前モデルを持ち、境界はコントラクトで統合
interface OrderPlacedEvent {          // 公表された契約（低強度）だけを共有
  orderId: string;
  totalAmount: { value: number; currency: string };
}
```

（他言語でも同等の考え方: 共有するのは契約＝スキーマ/DTOのみで、ドメインモデル本体は共有しない。）

## DRYの再解釈

排除すべき「重複」は**同一の揮発性知識**（同じビジネスルール、同じアルゴリズム）の複製であって、形が似ているだけの別知識ではない。逆に、たまたま似ている別ドメインの概念を共通化すると、無関係な変更理由が結合される（偶発的変動性の注入）。

## 実践指針

- 共通ライブラリへの追加基準を「便利そう」ではなく「低変動かつ汎用（generic相当）の知識か」にする。
- ドメインモデル（特にコアドメイン）は共有ライブラリに入れない。コンテキストごとに保持し、統合はコントラクト経由。
- 同一の揮発性知識の複製を発見したら1箇所へのカプセル化を優先する——重複は「暗黙の機能結合」で、明示的な依存より危険。
- 共有カーネルは同一チーム・近距離で、共有部分の変動を全員が即時吸収できる場合に限定する。
- 共有ライブラリの変更パターンを監視する: 1つの機能追加が常に「ライブラリ＋複数サービス」の同時変更になっているなら、高変動知識の共有シグナル（偶発的変動性）。知識の分配を見直す。
