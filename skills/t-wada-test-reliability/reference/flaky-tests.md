# flaky-tests — 信頼不能テスト

## 核心
信頼不能テスト（flaky test、古くはErratic Test）とは「**コード（プロダクト・テスト問わず）に一切手を触れていないのに、成功したり失敗したりする**」非決定的なテスト。t-wadaは『Googleのソフトウェアエンジニアリング』を引き、**「テストスイート全体の約1%が信頼不能になると、エンジニアはテスト結果を信じなくなり始める」**と繰り返し警告する。flakyの最大の害は個々の失敗ではなく、組織全体がテストの緑/赤を信用しなくなる「**テスト失敗への鈍感さ**」の伝播。

## 指針
- flakyと判明したテストには**「flaky」タグを付けてメインのCIラインから隔離**し、別ラインで動かす。安定して通るようになったらタグを外してメインへ戻す（**隔離→修復→復帰**）。
- テスト結果を**機械可読フォーマットで履歴保存**し、テストケースごとの信頼不能性を計測して優先度を判断する（連載第9回）。
- 非決定要因を設計で排除する（現在時刻・乱数・ネットワーク・DB・ファイル・非同期/並行処理・外部サービス依存を、テストダブルや注入で制御可能に）。
- E2E/Selenium系の**待ち時間依存**を、固定sleepでなく明示的な条件待ちに置き換える。
- flakyを安易にリトライで握りつぶさない。原因（共有資源・実行順序・並行性）の除去が本筋。

## 主な原因
- マルチスレッド安全性の欠如／共有資源への不適切な依存・テスト間の状態共有／外部資源（時刻・ネットワーク・DB・ファイル）の不適切な扱い／非同期・待ち合わせの不備／テスト実行順序への依存。
