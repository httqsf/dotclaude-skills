# fragile-tests — 脆いテスト

## 核心
脆いテストとは「**ソフトウェアを少し変えるたびに大量のテストを直さねばならない**」状態＝「手を触れるとすぐに失敗するテスト」。t-wadaはこれを偽陽性の一種と位置づける。根本原因は**テストコードとテスト対象の構造的結合度が高すぎる**こと。テストが名前空間・クラス・メソッド名・型といった構造や、内部アルゴリズム・実装の詳細に依存すると、外から見た振る舞い（公開API）は同じでも内部変更だけでテストが失敗し、**皮肉にもコードの変化・改善を妨げる**。

## flakyとの明確な違い
- **flaky** ＝ コードに手を触れていないのに成功/失敗が揺れる（非決定・不安定）
- **脆い** ＝ コードに手を触れるとすぐ失敗する（過結合・変更耐性の欠如）

## 指針（3原則）
- **公開API経由でテストする**（非公開＝privateに依存したテストは内部改善を妨げる）。
- **構造単位ではなく振る舞い単位でテストする**。
- **相互作用ではなく事後状態をテストする**（モックで相互作用を検証すると、些細な内部設計変更にモックを追随させ続ける必要が生じ変更コストが上がる）。
- モックを過剰に使わない。強力なテストダブルで無理にテストを書くと「現状の追認」になり、リファクタリング耐性を失う。
- 「テストが書きにくい＝設計が悪い兆候」（→ `t-wada-test-craft/reference/design-for-testability.md`）。

## アンチパターン
- モック過剰／相互作用テスト過多
- privateメソッド・内部構造への依存
- **モックドリフト**（テストダブルが本物の変更に追随せず乖離し、偽陰性を生む）
