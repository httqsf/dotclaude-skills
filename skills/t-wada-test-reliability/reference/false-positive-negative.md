# false-positive-negative — 偽陽性・偽陰性／自作自演

## 核心
偽陽性は「プロダクトコードが正しいのにテストが失敗する」現象、偽陰性は「プロダクトコードが誤っているのにテストが成功する」現象。t-wadaは**火災報知器の比喩**で説明する（火が出ていないのに鳴る＝偽陽性／火が出ているのに鳴らない＝偽陰性）。偽陰性は「テストできていると思っていたら、できていない」静かで恐ろしい問題。まず概念を正確に理解して問題を認識できることが上達の第一歩。
なお偽陰性は、狭義には「既存テストが誤って成功する」現象を指すが、広義には「**そもそも書かれていないテスト**（検証の欠落・カバレッジに映らないテスト不足）」も含めて語られる（`t-wada-test-strategy/reference/coverage-done-right.md` はこの広義）。

## 指針
- 偽陰性を検出するために**欠陥挿入法**を使う（プロダクトコードに意図的に明らかな誤りを混入し、テストがちゃんと失敗＝検知するか確認）。
- 期待値をプロダクトコードと同じロジックで計算しない。「**自作自演**」＝プロダクトとテストが同じ計算ロジックを共用すると、両方が同じ間違い方をしてテストが通ってしまう（税額計算の例）。期待値は独立に、手計算・別ロジック・ベタ書きで与える。
- テストが「**空振り**」していないか確認する（命名規則違反やアノテーション付け忘れでメソッドがテストとして認識されず未実行→常に緑）。
- skip/pending放置を監視する（無効化されたテストは守っているつもりで守っていない）。
- 信頼不能テストが失敗しても「またか、リトライすればいい」という**失敗への鈍感さ（狼少年化）**を招く。失敗には必ずアクションを取る文化を保つ。

## アンチパターン
- 自作自演（プロダクトとテストのロジック共用によるバグの相殺）
- 空振りテスト（実行されていないのに緑）
- skip/ignore放置
