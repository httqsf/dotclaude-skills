# purpose-driven-naming — 目的駆動名前設計 ★中核概念

## 核心
命名は単なるラベル付けではなく「**名前を設計し、ソフトウェアの構造や意図を明確に表現する立派な設計行為**」である。鍵は「存在ベースではなく**目的ベース**で名前を考える」こと。「商品」のような曖昧で広義な名前は **強力な心理的引力** を持ち、予約・注文・発送などあらゆるロジックを引きつけて神クラス（大きな泥団子）を生む。**目的ごとに特化した名前を与える**ことで関心が分離され、巨大クラスの登場を抑止し、クラス結合度が下がる。

## 実践指針
- **可能な限り具体的で意味範囲が狭い、特化した名前を選ぶ**: 「金額」より「請求金額」「消費税額」、「ユーザー」より「注文者」「受取人」。
- **目的ベース／役割ベースで発想**: 「そのモノは何のために存在するか」で名前を決める。存在ベース（〜Manager, 〜Processor, 〜Controller）は責務を曖昧にする。
- **技術駆動命名を禁止**: `Int〜`, `Str〜`, `〜Memory`, `tmp`, `flag`, `value01`, `cache` などプログラミング用語での命名はビジネス的意味を伝えず、関心の分離に貢献しない。
- **連番命名・略語を避ける**: `data01`, `data02` のような連番、意味を隠す略語は禁止。
- **ドメイン分析とユビキタス言語**: ユースケース分析・ドメインモデリングで関心事を洗い出し、ドメインエキスパートとの対話で用語を精緻化。利用規約からビジネス用語を抽出。
- **形容詞での状態区別を疑う**: 「仮の」「確定した」を形容詞で管理しているのは設計不足のサイン。別クラスに分けるべき兆候。
- **意味範囲の変化に警戒**: 仕様変更で言葉の意味が変わったら（個人顧客→法人顧客）名前自体を見直す。同じ「アカウント」でも業界（コンテキスト）で意味が異なるためコンテキスト単位で分割設計。
- **疎結合高凝集を点検**: 他概念との関連が多いほど結合度が高く悪い設計。「特化こそ正義、半端な汎用化は開発者の心身を滅ぼす」。

## コード例
```
// 悪い例: 存在ベース・広義な名前 → あらゆるロジックが集まり神クラス化
class Product { 予約() 注文() 発送() 在庫() 課金() ... } // 数千行に膨張

// 良い例: 目的ごとに特化・分離
class ReservationProduct { ... } // 予約品
class OrderProduct { ... }       // 注文品
class ShipmentProduct { ... }    // 発送品
// 値も目的特化: int money → class BillingAmount（請求金額）, class ConsumptionTax（消費税額）
```
