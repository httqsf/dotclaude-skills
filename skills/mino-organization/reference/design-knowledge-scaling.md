# design-knowledge-scaling — 設計知識の組織展開

## 核心
「大規模組織では専門家個人が全てをリファクタリングするのは不可能。スケールしない」。DMMでは自らは直接リファクタリングせず、「120人超のエンジニア一人ひとりが設計スキルを身につけ、日常的にリファクタリングできる状態」を作ることを目標にする。負債解消の本質は「何がどうなったら負債が解消された状態か（あるべき姿）」を定義することで、単なるコード改善でなく「利益を生み出し続けるソフトウェアの目指すべき形」を機能性・変更容易性の2つの内部品質として定義する。

## 実践指針
- **専門家は「実行者」でなく「増幅装置」になる**: 個人でリファクタリングせず、教育・ガイドライン・レビュー文化を通じて組織全体の設計力を底上げする。
- **設計責任者を各チームに育成する**: 組織の設計知識を分散配置し、レビューを通じて改善を横展開する。
- **設計ガイドラインを策定・文書化する**: 著書をベースに設計方針を明文化し、判断基準を組織で共有する。
- **レビューを教育の場にする**: PRレビュー時に本の具体例を参照したコメントを付けることで、レビューの効率化・一貫性向上・スキルアップを同時に実現する。
- **「あるべき姿」を先に定義してから負債に向き合う**: ゴール（機能性・変更容易性）を定義しないリファクタリングは的外れになる。
- **選択と集中**: メイン機能とサブ機能があればメインにコストを集中投下する。全面改善を一気にやろうとしない。
- **経営層とエンジニア双方の理解を前提にする**: 設計の価値を経営層にも理解してもらえる環境づくりが成功の前提。

## 具体的な取り組み例
### DMMでの施策（開発本部122人規模）
- **設計責任者の育成**: 約10%にあたる12人程度を「設計責任者」として育成。各チームのPRレビューを通じて設計改善を拡大。
- **モブ設計巡業（モブプログラミング巡業）**: 各開発チームを回ってモブプロを実施。スキルアップに効果的。
- **設計ガイドライン策定**: 著書『良いコード／悪いコード』をベースに設計方針を文書化。
- **チーム持ち回りの勉強会**: ワークショップ形式（`design-learning-workshop.md`）を全チームに展開。
- **リファクタリングの「言語化」**: 属人的になりがちな負債判断・改善判断を言語化し、レビューの一貫性とインシデント予防につなげる。

### READYFOR — 組織ビジョン「乳化」
- 「乳化」＝組織の中にエンジニアリングが自然に溶け込んでいる状態。異なる職能が混ざり合いイノベーションを促進。
- **Squad（職能横断チーム）体制**でエンジニアと非エンジニアを近接させ、「伝言ゲーム的な組織構造」による情報劣化を防止。
- 乳化はDDDと相補関係。ドメインエキスパートとエンジニアが協働してドメインモデルを構築し、「あるべき設計構造」を明確化する。

### キャリアから得た教訓（失敗談）
- READYFOR初期、コスト意識が不足したまま全面的リファクタリングを提案し現場から猛反発を受けてプロジェクトが頓挫。ここから「段階的・現実的なアプローチ」「選択と集中」の重要性を学んだ。
- 原動力は「設計が分からない人が多いのは嫌」「昔の自分を助けに行く」という使命感。
