# legacy-code-rescue — レガシーコード改善

## 核心
レガシーコードとは「**テストが無いコード**」（M.フェザーズの定義）。中がどれだけぐちゃぐちゃでも単体テストが整備されていればレガシーではない。だが「テストを整備するにはコードを変更する必要があり、コードを変更するにはテストが必要」という"レガシーコードのジレンマ"がある。**仕様化テスト（characterization test）**で現状の振る舞いを安全網として固定し、その上でリファクタして技術的負債を返済する。

## 指針
- 基盤整備の優先順位は **バージョン管理 → 自動化 → テスト**。早く入れるほどチーム全体にレバレッジが効く。
- いきなり網羅を狙わず、まず正常系（happy path）のテストから始める。
- テスト困難な既存コードには2つのアプローチ:
  - **Sprout（若葉）**: 既存を触らず、新規コードを独立させテスト付きで追加。
  - **Extract（抽出）**: 既存をテストで守りつつ切り出す。
- **仕様化テスト**＝「正しい仕様」ではなく「今の振る舞い」をそのまま写し取り、変化検知の網とする。
- 恒常的な危機モード（火消し）がリファクタ投資を妨げ、やがて生産性が崩壊する——日々のリファクタで予防する。
