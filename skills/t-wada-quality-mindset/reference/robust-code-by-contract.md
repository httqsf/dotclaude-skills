# robust-code-by-contract — 堅牢なコード

## 核心
「**対処 < 予防**」——問題への事後対処より、そもそも問題を起こさない予防が勝る（「予防に勝る防御なし」）。堅牢なコードは **予防的プログラミング → 攻撃的プログラミング（fail fast）→ 契約による設計（DbC）** の3段階で構築する。契約による設計では、**事前条件違反は呼び出し側の欠陥、事後条件違反は供給者側の欠陥**として責任の所在を明確化する。

## 3段階
- **予防（型による予防）**: 言語組み込み型より「ドメイン固有の型」を作り、「出来ていいことだけを出来る」ようにする。「正しく使う方が、操作ミスをするより簡単」な型を設計（きのこ本Tip「プリミティブ型よりドメイン固有の型を」）。
- **攻撃（fail fast / 表明）**: 暗黙の前提を `assert` で明示し、「起こり得ないこと」が起きたら速やかに落とす。表明はエラー処理には使わない。副作用のある式を assert に書かない。
- **契約（DbC）**: Hoare Triple `{P} A {Q}`（事前条件Pが成り立つときAを実行すると必ず事後条件Qが成り立つ）。「あなたが事前条件を満たして私を呼ぶなら、お返しに事後条件を満たすことを約束する」。

## 指針
- **正当性 vs 堅牢性の使い分け**: 「ミクロでは正当性を重視し、マクロでは堅牢性を重視する」。個々のクラスは fail fast（正当性重視）、堅牢性はアーキテクチャ/フレームワーク・グローバルハンドラ（catchして500画面等）で担保。
- **バグと例外を区別する**: プログラミングミス（＝バグ）はエラーハンドリングせず fail fast。運用時に起こり得るもの（＝例外）のみ処理する。
- 責任の公式: 「事前条件違反は呼び出し側に欠陥がある証拠」「事後条件違反は供給者側に欠陥がある証拠」。
