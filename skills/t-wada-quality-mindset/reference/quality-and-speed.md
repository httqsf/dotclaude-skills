# quality-and-speed — 質とスピード

## 核心
「品質を落とせば速く出せる」という命題は、犠牲にされる"品質"が何かを取り違えた誤解。速度と引き換えに犠牲にされるのは**内部品質（＝保守性）**であり、内部品質を落としても**短期的にすら速度は上がらない**。保守性の低さは欠陥・レビュー負荷・理解困難・実装の複雑化を生み、むしろ最初から遅くなる。質とスピードは対立ではなく相互に高め合う関係。

## 指針
- 磨くべきは「外部品質」（正しさ・使いやすさ・効率性・信頼性）ではなく、プロが左右できる「**内部品質**」（保守性・変更容易性・テスト容易性・理解容易性）。
- 「保守性を高めるのは数年後の誰かのためではなく、**1ヶ月後の自分**のためにやる」——投資は自分に返る。
- 「きれいなコードを書く方が、汚いコードを書くより（最初から）速い」（Bob Martin『Clean Architecture』を引用）。
- 質を落として得る"速度"は借金（技術的負債）であり、利息が付いて返ってくる。
- 本当のトレードオフは「質 vs スピード」ではなく、教育・設計・多様性への「組織の投資」にある。

## データ（損益分岐点）
- 内部品質への投資の損益分岐点は「**1ヶ月以内**」に現れる。
- テスト自動化の損益分岐点は「**約4回**」の実行で回収。
- Martin Fowler の Internal / External Quality の区別、"Is High Quality Software Worth the Cost?" を典拠として引用。

## 経営説得の骨子（そのまま文書化してよい）
1. 「品質を落とせば速くなる」で犠牲になるのは**内部品質（保守性）**である——外部品質（機能・正しさ）の話ではない
2. 内部品質を落としても**短期的にすら速くならない**（欠陥・レビュー負荷・理解困難で最初から遅くなる）
3. 投資の損益分岐点：内部品質は**約1ヶ月**、テスト自動化は**約4回**の実行で回収
4. 出典を添える：t-wada「質とスピード」講演、Martin Fowler "Is High Quality Software Worth the Cost?"
