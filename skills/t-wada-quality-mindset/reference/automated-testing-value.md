# automated-testing-value — 自動テストの価値

## 核心
自動テストの真の目的はコスト削減ではなく「**変化し続ける力を得ること**」＝躊躇なく変化に対応し続ける力。コスト削減目的で始めると、初期の学習コストと長期の保守負担が効果を上回り、結局は手動テストへ回帰して挫折する。目的は「信頼性の高い実行結果に短い時間で到達する状態を保ち、開発者に根拠ある自信を与え、ソフトウェアの成長を持続可能にすること」。

## 指針
- 目的を「コスト削減」ではなく「**持続的な変更・成長のためのインフラ**」と位置づける。
- 自動テストの信頼性を蝕む2つの「嘘」を排除する（→ `t-wada-test-reliability`）:
  - 偽陽性（火が無いのに鳴る）＝flakyテスト・実装に密結合した壊れやすいテスト。
  - 偽陰性（火が有るのに鳴らない）＝skip・カバレッジ不足・テストと実装の自作自演。
- 不確実性が約1%を超えると開発者はテスト結果を信頼しなくなる（Google の知見）。
- 「unit/integration」の曖昧な語より、Small / Medium / Large の「テストサイズ」で分類する（→ `t-wada-test-strategy`）。
- 「**テストのないコードはレガシーコード**」（M.フェザーズ）——どれだけ美しくカプセル化されていても、テストが無ければレガシー。
