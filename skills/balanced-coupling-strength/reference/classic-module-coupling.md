# classic-module-coupling — 構造化設計の古典的モジュール結合度

## 核心

Stevens・Myers・Constantineが1970年代の構造化設計方法論で提示した、モジュール間結合の強弱分類（内容/共通/外部/制御/スタンプ/データ）。Khononovはこれを「共有される知識の種類を初めて段階化した先行研究」と位置づけつつ、「レベルの一部は広すぎ、一部は細かすぎる」「用語がFortran/PL/I/COBOL時代のもの」と批判し、統合強度4レベルへ再編する**前提・原材料**として扱う。暗記対象ではなく、4レベルへの写像で現代語化して使う。

## 各レベル（coupling.devでのKhononov自身の整理、強い順）

1. **内容結合（Content / pathological coupling）**: 公開インターフェースではなく**内部の中身**が統合に使われる。暗黙的かつ脆い。例: 他モジュールの内部変数の直接書き換え、GOTOでの飛び込み。
2. **共通結合（Common）**: **共有メモリ空間**への読み書きで通信する。共有メモリの構造・保持情報・検証/業務ルールまで共有され、無関係なデータの理解も強制される。
3. **外部結合（External）**: 共通結合の限定版。共有メモリが**統合に関係する情報に厳密に限定**されるが、データの正しさを保証するルールの共有は残る。
4. **制御結合（Control）**: 一方が他方の**内部実行フローに影響を与える**。制御される側が実装詳細をカプセル化しきれていない。例: `process(data, mode)` のモードフラグ。
5. **スタンプ結合（Stamp）**: **データ構造**の形で情報を交換する。構造が実装詳細を露出し、不要なデータまで含む。例: `process(user)` が `id` しか使わないのに `user` 全体を受け取る。
6. **データ結合（Data）**: 統合に**必要な特定のデータのみ**を交換する最弱の理想形。

### 統合強度4レベルへの写像

内容→侵入、共通/外部/制御→機能、スタンプ→モデル、データ→コントラクト（詳細は `reference/integration-strength-four-levels.md`）。「外部結合→機能結合」の対応はcoupling.dev要約由来として提示（書籍本文での正確な対応は目安として扱う）。

## 実践指針

- 古典分類は「知識の種類」の初期分類として読み、4レベルへの写像で現代語化して使う。単体で設計判断に使わない。
- 制御フラグ引数（`processOrder(order, true, false, "express")`）を見たら制御結合＝呼び出し側が内部分岐を知っている兆候。意味ごとの関数分割（`processExpressOrder(order)`）で弱める。
- グローバル変数・共有ステート経由の連携は共通結合。共有範囲を統合関連情報に絞るだけでも外部結合まで一段弱まる。
- 「必要なデータだけ渡す」データ結合の方向は今も有効だが、Khononovはこれをコントラクト結合として精緻化している。本質は引数削減ではなく知識のカプセル化。

## コード例（説明用の創作例。他言語でも同等の考え方）

```ts
// 悪い例: 制御結合 — 呼び出し側が内部分岐を知っている
processOrder(order, true, false, "express");

// 良い例: 意味ごとに分割
processExpressOrder(order);
```
