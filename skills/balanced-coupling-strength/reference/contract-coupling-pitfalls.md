# contract-coupling-pitfalls — コントラクト結合の落とし穴

## 核心

コントラクト結合は「DTOを挟めば達成できる」ものではない。コントラクトの本質は知識のカプセル化（"a model of a model"）であり、**何の知識も隠さないDTOは強度を下げずに可動部分（moving parts）だけを増やし、システムをむしろ複雑にする**。Khononov本人が繰り返し警告する、統合強度モデルで最も誤用されやすいポイント。

## 本人の警告（趣旨。逐語保証なし）

- 「単に値をあるデータ構造から別のデータ構造へコピーしているだけなら、その抽象を導入しても**何の知識もカプセル化していない**」（Tech Lead Journal）
- 元モデルとDTOの属性が同一なら「**もう一つの可動部分**」を増やしただけ。認知負荷が上がり、モデル変更時に両方の更新が必要になる
- 判定原則: 「**抽象が知識のカプセル化に役立つなら有効。役立たないなら、追加の可動部分を導入するだけ**」（Tech Lead Journal）
- 「何の知識もカプセル化しない統合モデルは、システムをシンプルにしないばかりか、不要な『可動部分』を導入することでシステムをより複雑にしてしまう」（邦訳読者記事経由の書籍本文相当。Qiita/HrsUed、逐語保証なし）
- 知識を隠さない契約は「低強度を装いながら本質的な知識を漏らす」偽装になりうる（modularity/designスキル趣旨）

## 落とし穴のパターン

1. **鏡写しDTO**: 内部エンティティと1対1のフィールドを持つDTO。内部モデルの変更が毎回DTOと変換コードに波及＝実質モデル結合のまま、コード量だけ増加。
2. **変換層の増殖**: 画面ごと・エンドポイントごとに薄いDTO＋マッパーを量産し、変換処理と同期対象だけが増える。どこまで契約を分けるかは画面仕様の変動性とチーム間距離で判断すべき問題。
3. **契約に漏れる内部知識**: DTOに `internal_rank_score` やDB由来のステータスコードがそのまま載る＝名前を変えただけのモデル結合。静的コナーセンス（意味・位置）が残っていないか確認する。
4. **契約の安定性を管理しない**: コントラクト結合の価値は「モデルが変わっても契約の背後に変更を封じ込められる」こと。契約が内部都合で頻繁に変わるなら、消費者にとっては強い結合と同じ。
5. **カプセル化ではなく儀式としてのレイヤリング**: 「疎結合にしたい」だけの動機でイベント・DI・アダプタを積むのは、本質的結合（例: 税計算は小計に依存する）に偶発的複雑性を足す行為。

## 実践指針

- DTO導入時の判定質問: 「**この契約は内部モデルのどの知識を隠しているか**」を一文で言えるか。言えなければ導入しない（または内部モデル直渡し＝モデル結合を距離の近さで許容する）。
- 契約は消費者のニーズから設計する。内部モデルからの射影ではなく、利用文脈に必要な意味へ変換する: `user_status_code: 3` → `membership: "GOLD"`, `canPurchase: true`。
- 契約数の増加はコストとして計上する。変換層・同期・後方互換管理はすべて可動部分。低変動・近距離なら契約を挟まない選択が正しいことがある。
- バウンデッドコンテキスト越えは契約必須、コンテキスト内は不要な場合が多い、を目安にする（OHS・公表された言語・ACL）。
- 契約化した後も度合いを見る: 位置依存ペイロードやマジックナンバーが残れば静的コナーセンスとして強度が高止まりする。名前・型ベースへ弱める（`reference/connascence.md`）。

## コード例（説明用の創作例。他言語でも同等の考え方）

```ts
// 悪い例: 鏡写しDTO — 内部Customerと属性が1対1。何も隠していない
type CustomerDto = {
  id: string;
  rank: "NORMAL" | "GOLD";
  billingAddress: Address;
  creditLimit: number;
  internalScore: number;
};

// 良い例: 消費者（配送）のニーズから設計された契約
type ShippingDestination = {
  postalCode: string;
  address: string;
  recipientName: string;
};
```
