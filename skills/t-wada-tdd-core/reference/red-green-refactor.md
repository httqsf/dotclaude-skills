# red-green-refactor — 黄金の回転

## 核心
Red（失敗するテスト）→ Green（テストを通す最小実装）→ Refactor（テストを保ったまま設計改善）の3拍子のサイクル。t-wadaはこれを「黄金の回転」と呼ぶ。重要なのは**3本の矢印のうちリファクタリングへの矢印が最も弱い（最弱リンク）**という指摘。だからこそリファクタリングを特別なタスクにせず、**プログラミングの「名も無き1ステップ」として毎サイクルに組み込む**。

## 指針
- コードを変更・追加するときは**必ず一度Redにする**（「赤にするのがTDDの掟」）。ずっと緑だと「本当にテストが動いているのか」不安になる。
- Redを作るのは「**テストのテスト**」でもある。想定通り失敗することでテスト自体の正しさを確認する。
- リファクタリング＝「外部から見た振る舞いを保ちつつ、理解や修正が簡単になるように」コードを変える（Fowlerの定義）。
- リファクタリングのタイミングは**テストが通った直後**（テスト設計が頭に残っているうちに大胆に）。
- 重複が見えていなければ無理にリファクタしない。**重複が現れた所で初めて**行う。
