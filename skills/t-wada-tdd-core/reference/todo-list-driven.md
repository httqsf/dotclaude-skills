# todo-list-driven — TODOリスト/テストリスト駆動

## 核心
TDDの起点は、いきなりテストコードを書くことではなく、**これから実装する振る舞いを洗い出して「テストリスト（TODOリスト）」に書く分析ステップ**。t-wada訳のKent Beck「Canon TDD（TDDの定義）」では、このテストリスト作成が5ステップの第1歩。ここは「何を作るか」を考える分析の局面で、**実装設計の判断を混ぜてはいけない**。

## Canon TDD 5ステップ
1. **テストリストを書く** — 押さえたいテストシナリオ（期待される振る舞い）を洗い出す。分析の段階。
2. **1つだけ選ぶ** — リストから1つ選び、具体的・実行可能なテストコードにして失敗させる。
3. **テストを通す** — プロダクトコードを変更し、そのテストと過去の全テストを通す。
4. **必要に応じてリファクタリング** — ここで初めて実装の設計判断を行える。
5. **テストリストが空になるまで繰り返す**。

## 指針
- リストを**一度に全部テストコード化しない**（よくある間違い）。作業中に気づいた新しいケースは随時リストに足す。
- 「**最初に思ったものはだいたい間違っている**」。リストも実装もフィードバックで更新し続ける前提で持つ。
- 着手順は**テスト容易性が高い所から**。成功体験を積み勘所を育てる。
