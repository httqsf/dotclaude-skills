# tdd-as-design — 設計技法としてのTDD

## 核心
TDDは品質保証やテストカバレッジの手段ではなく、**分析・設計・開発を貫くプログラミングの技法**。t-wadaの決め台詞は「**TDDがもたらすのは"つくりやすい"設計ではなく"使いやすい"設計**」。テストを先に書く＝**使う側の期待からコードを逆算する**ため、利用者にとって使いやすいAPI・インターフェースが自然に導かれる。

## 指針
- テストを**最初の利用者（first client）**とみなす。作りやすさでなく**使いやすさ駆動**で設計する。
- 「正しい設計」は存在しない（「正しい設計というものはない、と考えたほうがよい」）。良い設計とは**ゴールの変化に応えられる**設計。
- 設計の良し悪しの信号は**テスト容易性**。3要素：**Observability（観測可能性）／Controllability（制御可能性）／Small（狭いスコープ）**。テストしにくさは設計の歪みのサイン（→ `t-wada-test-craft/reference/design-for-testability.md`）。
- TDDは「**開発者テストのTips集**」——Kent Beckが「開発者テスト（実装者が実装タイミングで書く自動テスト）」を漏れなくスムーズに進めるために具体化した方法論。
- 良いテストの最終評価軸は規約でもカバレッジでもなく「**根拠のある自信がつくかどうか**」。
