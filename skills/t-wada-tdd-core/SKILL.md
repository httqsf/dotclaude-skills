# t-wada-tdd-core — TDDの規律

t-wada（和田卓人）が日本に広めた **Kent Beck 由来のTDD** を、規律として実践する。ゴールは Ron Jeffries の「**動作するきれいなコード（clean code that works）**」。

> ⚠️ Red-Green-Refactor、仮実装・三角測量・明白な実装、Canon TDD の**原典は Kent Beck**。t-wada はその訳者・普及者。ここでは「t-wada が広めた Beck のTDD」として扱う。

## 貫く思想
「動作する」と「きれい」という2つの難問を**分割統治**する。青いルート（動作するコード→きれいにする）を選び、その障壁である**恐怖**には蛮勇でなく道具（バージョン管理・テスティング・自動化）で対抗する。TDDがもたらすのは「つくりやすい」でなく「**使いやすい**」設計。

## いつ・どう使うか（サイクル）

1. **テストリストを書く** → 実装する振る舞いを洗い出す分析ステップ（`todo-list-driven`）
2. **1つ選んでRed** → 具体的・実行可能なテストで失敗させる
3. **Green** → 不安の大きさに応じて仮実装/三角測量/明白な実装を使い分け最小実装（`fake-triangulate-obvious`）
4. **Refactor** → テストを通したまま重複除去（`red-green-refactor`）。最弱リンクなので毎回意識する
5. リストが空になるまで繰り返す

テストコードの書き方は `t-wada-test-craft`、テスト戦略は `t-wada-test-strategy` へ。

## クイック原則
- 変更・追加時は**必ず一度Redにする**（「赤にするのがTDDの掟」）。Redは「テストのテスト」
- リファクタは特別なタスクでなく「**名も無き1ステップ**」。テストが通った直後に行う
- 「**最初に思ったものはだいたい間違っている**」——リストも実装もフィードバックで更新
- テストを**最初の利用者（first client）**とみなす
- 良いテストの最終評価軸は「**根拠のある自信がつくか**」

## エージェント運用規則
- テストを書いたら**必ず実行して失敗（Red）を確認**してから実装に進む。書いただけで実装に進まない
- 各サイクルの終わりに**全テストを実行**する（そのテストだけ通して満足しない）
- テストリスト（TODO）は着手前にユーザーへ提示し、大きな仕様判断はリスト段階で合意する
