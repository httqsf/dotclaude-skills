# purpose-goal-means — 目的・目標・手段

## 核心
「システムとは目的達成手段である」を根本前提に置く。二足歩行も自動車もソフトウェアも、すべて「〜〜したい」という目的を叶える手段。ゆえに設計は**目的から**考える。目的が違えば課題も解決手段（コード・クラス設計）も違う。従来分離されがちな「要求定義」と「モジュール設計」は、「構造化は目的単位である」という一点で地続きになる。

## 3層の定義（本人の言葉）
- **目的**：目指すべき状態、行動のねらい、目当て。抽象的。＝顧客の要求に相当
- **目標**：何をもって目的達成とみなすかを示す具体条件。定量検証可能。＝機能要件・仕様・制約に相当
- **手段**：目的を達成するための方法や道具。＝ソースコード・クラス設計に相当

## 実践指針
- 仕様書を渡されて「単に仕様を満たす／動かすことがゴール」で止めない。仕様の背後の**目的**を掴む。
- 異なる目的のロジックは**目的ごとに別クラスへ分離**する（＝関心の分離）。
- ドメインエキスパートとは「**事業目的と、目的達成上の課題**」に焦点を当てて議論する。目的が決まって初めて課題が浮き彫りになる。
- AIへの指示でも**目的を伝えると回答精度が上がる**。目的駆動で指示すればAIも目的ごとにクラスを分離できる。
- 命名は目的ベースで（「意図の明白なインターフェース」＝効果と目的を記述し、手段には言及しない）。
- 契約による設計（事前条件・事後条件・不変条件の明記）で目的達成の正確性を保証する。

## 具体例
- **ポイントボーナス計算**：一般会員（1%）とゴールド会員（2%）の計算を単一クラスに混在させると、ゴールド会員側の仕様変更が一般会員側にバグを波及させる → 「一般会員用」「ゴールド会員用」で目的ごとにクラス分離。
- 比喩：目的の違うロジックの混在は「フライパンとトイレ用ラバーカップを同じ道具箱に混在させる」ようなもの。しかもソースコードは「味も臭いもしないので判断がつきにくい」。
