# quality-attributes — 品質特性

## 核心
利益を生み出し続けるソフトウェアには「**機能性（目的を達成できる）**」と「**変更容易性（目的の変化に対応できる柔軟性）**」の**両立**が必要。ISO/IEC 25010の品質特性群の中でも、継続的・持続的開発においては保守性の副特性である**変更容易性（modifiability）を最重要**と位置づける。「技術的負債とは『あるべき姿』と『今』の差」であり、その本質は変更容易性の低さ。

## 変更容易性の定義（本人の言葉）
「**仕様変更時、なるべくバグを埋め込まず、素早く正確にロジック変更できるかを指し示す度合い**」

## ISO/IEC 25010との関連
- 保守性（Maintainability）の副特性＝モジュール性・再利用性・解析性・修正性（変更容易性）・試験性。この中の**修正性/変更容易性**を軸に据える。
- 品質特性を「良いコード＝保守性が高い／悪いコード＝保守性が低い」の評価軸として使う。
- ビジネス目的の設計では特に「機能適合性（機能性）」と「変更容易性」が効く。

## 実践指針
- 変更容易性が低いコードの弊害を認識する：バグを埋め込みやすい／変更に何日もかかる／どこかを直すと別の箇所でバグる／影響箇所の特定に莫大な労力。
- **自己防衛責務**を持たせ「**常に正常なインスタンスのみが存在可能な構造**」を目指す（不正値を作れないクラス設計）。
- 技術的負債の解消では、まず「**何がどうなったら負債が解消された状態か**（あるべき姿）」を定義する。
- 設計は一度で完成しない。実装と並行して**何度も作り直し、少しずつ理想構造に近づける**。
- アプリケーションとDBの関心を分離する（Repository / QueryServiceパターンでDBロジックを隔離）。
