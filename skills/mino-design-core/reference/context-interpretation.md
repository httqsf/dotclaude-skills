# context-interpretation — 文脈の解釈

## 核心
最重要洞察の一つが「**あらゆる目的に対応可能な、たったひとつの万能な商品モデルはありえない**」。同じ「商品」という概念でも、**目的（文脈・関心事）が違えば全く別のモデルになる**。目的が違えば解決すべき課題が違い、モデルが備えるべきデータと振る舞いが違うから。DDDの「境界付けられたコンテキスト」を、独自の「目的−抽象化」の切り口で再解釈したもの。

## 抽象化の定義（本人の言葉）
- 問題領域における抽象化＝「対象物に付随する様々な特徴のうち、**目的達成する上で邪魔になる課題を抜き出すこと**」
- 連鎖：「**目的が決まる → 課題が浮き彫りになる → 解決手段が決まる**」

## 実践指針
- 同じ言葉でも**コンテキストが異なれば意味が異なる**（目的が違うから）。ユビキタス言語は「目的を識別・共有するための言葉」。単なる用語集は目的が分からずモデリングに役立たない。
- テーブル/モデルを複数目的で使い回し、**文脈によってカラムやデータの意味が変わる構造を設計しない**（＝単一責任原則の実践、「単一責任原則とは単一目的原則」）。
- **DRY原則はコードの重複ではなく「意図や目的の重複」を許さない原則**。見た目が同じロジックでも目的が違えば共通化しない。
- 凝集度＝「ある**目的**におけるデータとロジックの関係性の強さ」。目的ごとに高凝集、目的が異なるものは疎結合に。
- 命名を目的特化に：「商品」ではなく「審査品目」「在庫品」「注文品」「配送品」。

## 具体例（同じ「商品」が文脈で変わる）
- **審査コンテキスト**＝「審査品目」：品目分類・審査所見・審査結果
- **在庫管理コンテキスト**＝「在庫品」：発注金額・在庫量・安全在庫量
- **注文コンテキスト**＝「注文品」：取扱開始日・売値・在庫量
- **配送コンテキスト**＝「配送品」：サイズ・重量

## アンチパターン
- **一枚岩モデル**：全目的対応の単一商品モデルは、どの状況で何のデータが使われるか不明／不変条件が状況で異なり振る舞いが不明確／言葉やデータが多義的になり混乱・エンバグ／密結合で影響範囲分析に膨大な労力。
- **DRYの誤用**：通常割引5%と夏季割引5%が同じロジックだからと割引率を共通化してはならない（一方の要件変更が他方に波及する）。
- **ピザ注文モデル**：単に「注文モデル」を作るのでなく、「調理キャパを超えない」「食材在庫を超えない」「注文数が−1にならない」といった課題を「追加()」「削除()」メソッドと制約条件としてカプセル化する。→「モデルはデータと判断/計算ロジックをカプセル化して課題解決要素を備えた目的達成手段になる。これがDB設計のデータモデルとDDDのドメインモデルの決定的な違い」。
