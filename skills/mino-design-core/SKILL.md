# mino-design-core — ミノ駆動設計の根幹思想

ミノ駆動（仙塲大也、『良いコード／悪いコードで学ぶ設計入門』著者）の設計思想の**背骨**を提供する。個別テクニック（命名・カプセル化・リファクタリング）はすべてここから派生する。迷ったら常にここへ戻る。

## 貫く一本の連鎖（最重要）

```
目的(purpose) → 課題の抽出(＝問題領域の抽象化) → 目的達成手段(means＝コード/設計)
              ↓ 目的が違えば手段も違う
        文脈(context)ごとにモデルが分かれる → 変更容易性(quality)が生まれる
```

この連鎖を無視した「一枚岩／目的混在」こそが**技術的負債の正体**。あらゆる設計判断は「これは何の目的を達成する手段か？」から始める。

## いつ・どう使うか

設計判断に迷ったら、以下の3つの問いを順に当てる:

1. **目的は何か？** → この仕様・機能・クラスは誰の何を叶えるための手段か。仕様を満たすだけで止めない。
2. **どの品質を守るか？** → 継続開発では原則 **変更容易性** を最優先。「バグを埋め込まず素早く正確に変更できるか」で判断。
3. **文脈は一つか？** → 同じ言葉（商品・ユーザー・金額）でも目的が違えば別モデル。「万能な単一モデルはありえない」。

このスキルは他の `mino-*` スキルの**前提**。ドメイン設計は `mino-domain-design`、クラス/メソッド実装は `mino-code-design`、負債返済は `mino-refactoring`、組織展開は `mino-organization` へ。

## クイック原則（本人語彙）

- 「システムは**目的達成手段**」— 目的から設計を考える
- 「**単一責任原則＝単一目的原則**」— 1クラス1目的
- 「**DRYは目的の重複を許さない**」— 見た目が同じでも目的が違えば共通化しない（通常割引5% と 夏季割引5% を共通化するな）
- 「常に**正常なインスタンスのみが存在可能な構造**」を目指す
- 「万能な商品モデルはありえない」— 目的ごとにモデルを分ける
- 「基本こそ奥義」

## アンチパターン検知（これを見たら赤信号）

- 仕様を満たすことだけがゴールになっている（目的が不在）
- 1つのクラス/テーブルが複数の目的で使い回され、文脈でカラムの意味が変わる
- 「後で直す」で変更容易性を犠牲にしている
- 目的が違うロジックが同一クラスに混在（＝フライパンとラバーカップを同じ道具箱に入れている）
