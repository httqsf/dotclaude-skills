# test-ratio-migration — 比率の是正：サイズダウン戦略

## 核心
既存の Large 偏重（アイスクリームコーン）を、**Large → Medium → Small へ段階的にサイズダウン**して望ましい比率に近づける。主眼は「**いきなりテストダブルを作るな**」——まず実物がテスト制約内で動くか／設計改善でサイズを下げられないかを検討し、それでも必要なときだけ信頼できるダブルを使う。テストダブルは忠実性と決定性のトレードオフで、多用は「脆いテスト」「偽陰性」を生む。

## 3段階（第6回「サイズダウン戦略」）
- **段階1: テストダブルを避ける** — 実物がテスト制約に収まるか確認。公式コンテナイメージ（MySQL/Redis等）なら **Medium** で済む。**設計を疎結合化**（Humble Object：ロジックをテストしやすい純粋部分へ抽出し、外部依存の薄皮だけ残す）すれば Small で書ける範囲が広がる。
- **段階2: テストダブルを自作しない** — 必要なら「**信頼できる fake**」を探す。公式提供（AWS 推奨の **DynamoDB Local**）やコミュニティ検証が効くOSS（**LocalStack**, Moto 等）を優先。
- **段階3: 慎重に自作する** — チーム間はスキーマ固定、**Consumer-Driven Contract（Pact）**、外部通信の記録再生（**VCR** 系）を活用。
- **E2E / Large は CUJ（Critical User Journey）に絞る** — 全機能をE2Eで網羅せず、重要なユーザー体験の道筋だけを担保。

## 望ましい比率（目安）
- 初期目標: **Small : Medium : Large = 70 : 20 : 10**
- 成熟・大規模化に伴い: **80 : 15 : 5** を志向。
