# coverage-done-right — カバレッジを目的化しない

## 核心
カバレッジは**目的ではなく道具（味方）**。「**カバレッジ100%は100点ではない**」。カバレッジが測るのは「コードのうちどれだけ実行されたか」であって「仕様をどれだけ満たすか」ではない。バグの多くは「テストされていないコード」より「**そもそも書かれていない（＝仕様の理解不足による）テスト**」＝偽陰性／テスト不足から生まれ、これはカバレッジには映らない。

## 指針
- カバレッジを**管理・報告のための数値目標にしない**（「管理されるカバレッジ」でなく「自分の開発を助けるカバレッジ」へ）。
- 「**カバレッジに使われるな、カバレッジを使え**」——開発中のフィードバック（未到達コードの発見）として使う。
- 100%を追わない。**90%前後までは価値が大きい**が、そこから100%へは労力が不釣り合い（収穫逓減、黒魔術的ハックになりがち）。
- 見るべきは「コードのカバレッジ」より「**仕様のカバレッジ**」——重要な仕様・振る舞いが検証されているか。
- カバレッジが低いうちは測定効果が大きい。低いところを優先的に埋める。

## 比喩
- **体重計**：「テストは体重計みたいなもの。**ただ乗っただけでは痩せない**」。カバレッジ（測定）は現状を映すだけで、質を上げるのは**設計改善という行動**。
- 「**右手に感情、左手に数値**」——数値だけでも感情だけでもなく両方を携えてコードに向き合う。
