# test-pyramid — テストピラミッドとSMURF

## 核心
テストモデルが乱立して見える（ピラミッド／アイスクリームコーン／テストトロフィー／ハニカム）のは、**土台の「ユニット/統合/E2E」の定義が人により違うから**。これらを**テストサイズの軸で並べ直すと、どれもおおむねピラミッド型に収束する**。本人の言葉「テストスコープで分類するとトロフィー型になるが、テストサイズで分類するとピラミッド型になる」。

## 指針
- モデル論争（トロフィー vs ピラミッド等）に囚われず、**まず自分たちのテストをサイズで測り直す**。
- 目指す姿は「**テストの大半を、統合環境を必要とせずに実施できる**こと」＝Smallが土台。
- **アイスクリームコーン（E2E過多）は避ける**（遅く・不安定・原因特定困難）。
- **砂時計型（中段が薄い）も避ける**（全体の安定性が落ちる）。
- ピラミッドの目的は形そのものでなく「**自動テストの信頼性を中長期的に保つ**」こと。

## 評価軸 SMURF（Google）
テストサイズのトレードオフを可視化する5観点：
- **S**peed（速度）／**M**aintainability（保守性）／**U**tilization（リソース使用効率）／**R**eliability（信頼性）／**F**idelity（本番への忠実性）
- サイズを上げると Fidelity は上がるが Speed/Reliability が下がる。
