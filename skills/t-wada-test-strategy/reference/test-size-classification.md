# test-size-classification — テストサイズ分類

## 核心
「ユニットテスト／統合テスト／E2E」という**スコープベースの分類は人によって定義がブレる**ため議論が発散する。そこで Google 由来の「**テストサイズ**」＝テストが**どこまでのリソースにアクセスするか**という機械的・客観的基準で Small / Medium / Large に分類する。**速度と決定性（安定性）はサイズの結果として決まる**。

## 定義境界

| 項目 | Small | Medium | Large |
|---|---|---|---|
| プロセス | 単一プロセス内 | 単一マシン内（複数プロセス可） | 複数マシンにまたがる |
| ネットワーク | 禁止（localhost含め不可・ダブルで代替） | localhostのみ許可 | リモート許可 |
| データベース | 禁止（ダブルのみ） | ローカル（コンテナ等）許可 | 本物許可 |
| ファイルシステム | 禁止 | ローカル許可 | 許可 |
| マルチスレッド/sleep | 原則不可 | 許可 | 許可 |
| 特徴 | 速い・決定的 | 中程度 | 遅い・flakyになりやすい |

「single process → single machine → 制約なし」という入れ子構造が本質。

## 指針
- 分類は主観（「単体か結合か」）でなく、**アクセス範囲という観測可能な事実**で機械的に決める。
- **Smallを最優先**で書く（速い・安定・並列化しやすい・失敗箇所を特定しやすい）。
- ネットワークアクセスは「一番遅くて不安定」なので、Smallではテストダブルで排除する。
- CIパイプラインでは **Small→Medium→Large の順に実行**し、早く安く失敗を捕まえる。
- サイズをタグ付け／ディレクトリ分けし、実行環境（並列度・タイムアウト）を分ける。
